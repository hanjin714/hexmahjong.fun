# 海克斯麻将 H5 玩法白皮书

版本：H5 原型规则 v0.3  
日期：2026-05-30  
适用项目：`HexMahjongH5`

## 1. 产品定位

海克斯麻将是一款把麻将、城邦规则、海克斯强化、羁绊成长融合在一起的轻策略牌局游戏。它保留麻将“摸牌、出牌、成型、胡牌”的核心节奏，同时引入类似自走棋的阶段强化和全局规则变化，让每一局都有不同的运营路线。

当前 H5 版本是可运行原型，目标是验证核心玩法、UI 反馈、移动端交互和未来微信小游戏适配方向。

## 2. 核心循环

1. 选择城邦：确定整局的全局规则。
2. 选择海克斯：如果城邦允许海克斯，玩家在阶段节点三选一。
3. 发牌开局：4 名玩家参与，玩家为 0 号位，其余为 AI。
4. 回合进行：当前玩家摸 1 张，再出 1 张。
5. 玩家操作：可出牌、过牌托管、使用可用海克斯、满足条件时胡牌。
6. 阶段强化：第 1、3、6 轮触发海克斯选择，最多持有 3 个。
7. 胜负结算：胡牌或牌山耗尽后进入结算反馈。

## 3. 牌与基础参数

### 3.1 花色

当前 H5 使用三种基础花色：

| 花色 ID | 中文显示 | 颜色定位 |
|---|---|---|
| `character` | 筒 | 黑灰 |
| `bamboo` | 条 | 绿色 |
| `wan` | 万 | 蓝色 |

### 3.2 基础牌组

每种花色包含 1 到 9，每张 4 份。

基础牌数：

```text
3 种花色 * 9 个点数 * 4 份 = 108 张
```

### 3.3 混子

默认混子数量为 2 张。混子在当前 H5 规则中用于胡牌判定补缺：

- 可补缺失点数。
- 可补对子。
- 可补刻子。
- 不计入普通花色集合。

混子数量受以下规则影响：

| 来源 | 效果 |
|---|---|
| 基础规则 | 2 张混子 |
| 祖安实验室 | 混子数量乘以 2 |
| 混子成双 | 创建牌组时混子数量再乘以 2 |
| 对局中获得混子成双 | 额外向牌山加入 3 张混子 |

## 4. 城邦系统

城邦是第一层规则，开局选择后影响整局。

| 城邦 | 当前 H5 效果 | 当前实现状态 |
|---|---|---|
| 净水监狱 | 传统麻将规则，无海克斯选择 | 已实现 |
| 皮尔特沃夫金库 | 描述为每轮获得基础筹码 | 已展示，筹码经济未完全实装 |
| 祖安实验室 | 混子数量翻倍 | 已实现 |
| 艾欧尼亚道场 | 只能胡清一色类牌型 | 已实现 |
| 诺克萨斯角斗场 | 点炮惩罚翻倍 | 已展示，点炮结算未完全实装 |
| 班德尔城传送门 | 每 3 轮随机交换一张手牌 | 已实现玩家和随机 AI 交换 |

### 4.1 净水监狱

效果：

- 不进入海克斯选择。
- 选中后直接开局。

设计目的：

- 给新玩家提供低复杂度入口。
- 作为无强化环境下的基准玩法。

### 4.2 皮尔特沃夫金库

当前效果：

- 作为海克斯可用城邦。
- UI 与规则描述中保留“每轮基础筹码”概念。

当前限制：

- H5 原型尚未完整接入筹码经济。

### 4.3 祖安实验室

效果：

- 创建牌组时混子数量乘以 2。

策略影响：

- 提高胡牌概率。
- 同时提高 AI 胡牌概率。
- 更适合追求高波动和快速成型。

### 4.4 艾欧尼亚道场

效果：

- 如果非混子牌包含多个花色，则胡牌判定直接失败。

策略影响：

- 强制玩家朝单一花色运营。
- 对海克斯“千丝增幅器、铜墙铁饼、万法归宗”价值提升明显。

### 4.5 诺克萨斯角斗场

当前效果：

- 城邦已存在并展示。
- `discardPenalty` 作为规则字段存在。

当前限制：

- 当前 H5 尚未实现完整点炮/筹码赔付，所以翻倍惩罚暂未进入实际结算。

### 4.6 班德尔城传送门

效果：

- 每当轮数为 3 的倍数时触发。
- 玩家随机与一名 AI 交换 1 张手牌。

触发逻辑：

```text
如果 city.exchange == true 且 round % 3 == 0，则触发交换
```

## 5. 海克斯系统

海克斯是第二层个人强化。当前 H5 采用自走棋式阶段节奏。

### 5.1 选择节奏

| 阶段 | 规则 |
|---|---|
| 第 1 轮 | 开局三选一 |
| 第 3 轮 | 阶段奖励三选一 |
| 第 6 轮 | 阶段奖励三选一 |

限制：

- 最多持有 3 个海克斯。
- 已选择的海克斯不会再次出现在候选池。
- 每次从剩余海克斯中随机抽 3 个展示。

### 5.2 海克斯列表

| 海克斯 | 品级 | 类型 | 当前 H5 效果 |
|---|---|---|---|
| 饼干海克斯 | 棱彩 | 主动 | 将条/筒中点数 3 以上的牌拆成 3 张 |
| 混子成双 | 黄金 | 被动 | 增加混子数量 |
| 千丝增幅器 | 白银 | 资源 | 立即获得 3 张条 |
| 铜墙铁饼 | 白银 | 资源 | 立即获得 3 张筒 |
| 万法归宗 | 白银 | 资源 | 立即获得 3 张万 |
| 活力再生 | 黄金 | 主动 | 弃 2 张孤牌，摸 2 张，冷却 2 轮 |
| 全能吸血 | 黄金 | 反应 | 当前提供羁绊路线展示，实际拿牌反应未实装 |

### 5.3 饼干海克斯判定

可用条件：

- 当前是玩家回合。
- 已拥有饼干海克斯。
- 已选择一张手牌。
- 该牌不是混子。
- 该牌不是万。
- 该牌点数大于等于 3。

拆分结果：

```text
原牌 X -> 同花色 1 + 同花色 1 + 同花色 (X - 2)
```

示例：

```text
条5 -> 条1、条1、条3
筒9 -> 筒1、筒1、筒7
万5 -> 不可拆
混 -> 不可拆
```

UI 反馈：

- 可拆牌在手牌上显示“可拆”。
- 选中可拆牌后显示“将拆”。
- 底部按钮显示具体变化，如 `条5 -> 条1、条1、条3`。

### 5.4 活力再生判定

可用条件：

- 当前是玩家回合。
- 已拥有活力再生。
- 距离上次使用至少 2 轮。
- 玩家手中至少有 2 张非混子牌。
- 牌山至少有 2 张牌。

弃牌选择：

- 当前 H5 自动选择排序后最靠前的 2 张非混子牌。
- 这些牌会进入弃牌堆。
- 然后玩家从牌山摸 2 张。

UI 反馈：

- 预计弃掉的 2 张牌会在手牌上显示“将弃”。
- 底部提示显示如 `弃 筒2、筒2 -> 摸 2 张`。

### 5.5 资源类海克斯判定

资源类海克斯包括：

- 千丝增幅器：获得 3 张条。
- 铜墙铁饼：获得 3 张筒。
- 万法归宗：获得 3 张万。

生效逻辑：

- 如果开局选择，则发牌后立刻从牌山中取对应花色的非混子牌加入玩家手牌。
- 如果中途阶段选择，则选择后立即生效。
- 若牌山中对应花色不足，则能取几张取几张。

## 6. 回合系统

### 6.1 玩家结构

当前 H5 固定 4 名玩家：

| index | 名称 | 控制 |
|---|---|---|
| 0 | 你 | 真人玩家 |
| 1 | AI 1 | 自动 |
| 2 | AI 2 | 自动 |
| 3 | AI 3 | 自动 |

### 6.2 开局发牌

开局流程：

1. 创建牌山。
2. 洗牌。
3. 每名玩家发 13 张。
4. 应用开局资源类海克斯。
5. 玩家摸 1 张，进入玩家回合。

### 6.3 回合推进

每个回合：

```text
当前玩家摸 1 张 -> 当前玩家出 1 张 -> 进入下一家
```

当 `currentPlayer` 从 3 回到 0：

- 轮数 `round` 加 1。
- 检查城邦事件。
- 检查是否进入海克斯选择阶段。

### 6.4 玩家倒计时

玩家回合默认倒计时：

```text
18 秒
```

倒计时结束：

- 如果玩家选中了手牌，自动打出选中牌。
- 否则由 AI 弃牌策略选择一张自动打出。

倒计时最后 3 秒：

- 播放 tick 音效。
- 计时环变色提醒。

### 6.5 AI 出牌策略

AI 简化策略：

1. 先检查自身是否满足胡牌条件，且分数大于等于 100。
2. 如果可胡，则直接胡牌。
3. 否则优先丢孤张。
4. 若无孤张，则丢排序后的第一张非混子牌。
5. 若都没有，则丢手牌第一张。

孤张定义：

- 同花色前后没有相邻点数。
- 同一张牌数量为 1。

## 7. 胡牌判定

### 7.1 总入口

胡牌入口为 `checkWin(hand)`。

基础要求：

```text
手牌数量 >= 14
```

如果当前城邦是艾欧尼亚道场：

```text
非混子牌必须只有一种花色，否则不能胡
```

### 7.2 胡牌牌型

当前 H5 按以下顺序判定，命中后立即返回：

1. 清一色大成
2. 七对子
3. 四刻成型
4. 偏门成势

### 7.3 清一色大成

条件：

- 非混子牌只有一种花色。
- 手牌中需要覆盖该花色 1 到 9。
- 缺失点数可由混子补足。

分数：

```text
260 + 羁绊分
```

示例：

```text
条1、条2、条3、条4、条5、条6、条7、条8、条9 + 其他条/混子
```

### 7.4 七对子

条件：

- 牌面可组成至少 7 个对子。
- 每两个相同非混子牌算 1 对。
- 每两个混子可算 1 对。

分数：

```text
140 + 羁绊分
```

### 7.5 四刻成型

条件：

- 牌面可组成至少 4 个刻子。
- 3 张相同非混子牌算 1 刻。
- 2 张同牌 + 1 混子可补成 1 刻。
- 1 张单牌 + 2 混子可补成 1 刻。

分数：

```text
120 + 羁绊分
```

### 7.6 偏门成势

条件：

- 手牌数量至少 14。
- 某一花色的非混子牌数量 + 混子数量 >= 10。

分数：

```text
90 + 羁绊分
```

设计目的：

- 给尚未完全成型但明显朝一个方向运营的手牌提供低门槛胡牌出口。
- 提高 H5 原型的胡牌频率，便于测试胜负结算。

## 8. 羁绊系统

羁绊是第三层奖励，胡牌时附加分数。

### 8.1 顶级花色羁绊

| 羁绊 | 条件 | 分数 |
|---|---|---|
| 万法归一 | 万牌数量至少 11，且包含 1 和 9，缺失可由混子补 | +1000 |
| 千丝万缕 | 条牌数量至少 11，且包含 1 和 9，缺失可由混子补 | +1000 |
| 圆满无缺 | 筒牌数量至少 11，且包含 1 和 9，缺失可由混子补 | +1000 |

### 8.2 普通羁绊

| 羁绊 | 条件 | 分数 |
|---|---|---|
| 三花聚顶 | 万、条、筒中至少 3 个花色各有 2 张以上 | +50 |
| 四喜临门 | 至少 4 个对子 | +30 |

### 8.3 羁绊与胡牌关系

- 羁绊不会单独结束游戏。
- 羁绊在胡牌判定时计算。
- 胡牌总分 = 基础牌型分 + 所有触发羁绊分。

## 9. 操作按钮与 UI 判定

### 9.1 按钮显示规则

| 按钮 | 显示条件 |
|---|---|
| 胡牌 | 当前玩家手牌满足胡牌判定 |
| 过牌 | 当前是玩家回合 |
| 活力 | 当前满足活力再生可用条件 |
| 拆牌 | 当前满足饼干海克斯可用条件 |
| 重开 | 对局中或结算后可用 |

### 9.2 操作提示

当前 H5 提供两层提示：

1. 底部提示条：说明当前最重要操作的结果。
2. 手牌高亮：标记“可拆、将拆、将弃”等牌面状态。

优先级：

```text
可胡提示 > 可拆提示 > 活力提示
```

### 9.3 胡牌反馈

胡牌按钮特点：

- 金色高亮。
- 红点提示。
- 脉冲动画。
- 显示牌型和分数。

胡牌后：

- 弹出胜利/惜败结算层。
- 展示牌型、得分、触发羁绊、最终手牌。
- 播放胜利或终局音效。
- 播放粒子和光效动画。

## 10. 音效系统

当前 H5 使用 WebAudio 程序化生成音效，不依赖外部音频文件。

| 事件 | 音效 |
|---|---|
| 选择按钮 | 短促点击音 |
| 摸牌 | 轻提示音 |
| 出牌 | 噪声 + 低频点击 |
| AI 出牌 | 低音短提示 |
| 使用技能 | 上扬三音 |
| 海克斯出现 | 阶段提示音 |
| 选择海克斯 | 上扬四音 |
| 倒计时最后 3 秒 | tick 音 |
| 胡牌胜利 | 上扬胜利音阶 |
| 流局/失败 | 下行终局音 |

支持浏览器振动能力的移动设备上，选择、出牌、技能、海克斯、倒计时和胡牌会触发轻量触感反馈。

## 10.1 牌面视觉

当前 H5 采用 Canvas 程序化牌面：

- 筒子：圆点图案。
- 条子：竹节图案。
- 万字：大号数字与“万”字。
- 混子：金色牌面，并显示具体底牌，例如 `混(筒5)`、`混(条5)`。
- 新摸牌：显示“新”标签与蓝色光效。

后续如需切换真实贴图，建议优先使用授权清晰的公有领域或开源资源，并在项目中保留来源与许可证说明。

## 11. 结算系统

### 11.1 胜利

玩家胡牌后：

- 显示“胜利”。
- 显示玩家牌型。
- 显示正向分数。
- 展示最终手牌。
- 展示羁绊标签。

### 11.2 惜败

AI 胡牌后：

- 显示“惜败”。
- 显示胡牌 AI 与牌型。
- 展示 AI 最终手牌。
- 展示触发羁绊。

### 11.3 流局

当牌山为空时：

- 对局进入 `gameOver`。
- 状态显示“流局”。
- 当前版本没有单独流局结算面板，可作为后续补充。

### 13.2 核心模块

## 14. 后续版本建议

### 14.1 规则完善优先级

1. 补齐吃、碰、杠。
2. 补齐点炮、自摸、放炮责任。
3. 完整接入筹码经济。
4. 实装全能吸血。
5. 增加流局结算。

### 14.2 体验优化优先级

1. 增加听牌提示。
2. 增加可胡牌来源说明。
3. 增加海克斯选择前后的路线推荐。
4. 增加结算复盘。
5. 增加设置面板：音量、动画强度、回合时间。

### 14.3 小程序适配建议

当前 H5 已经把主要牌桌绘制集中在 Canvas，但仍使用 DOM 处理 HUD、弹层和按钮。迁移微信小游戏时建议：

1. 保留 `game.js` 状态机和判定逻辑。
2. 将 DOM 按钮迁移为 Canvas 或小程序组件。
3. 将 WebAudio 替换为微信小游戏音频 API。
4. 将图片资源压缩为小游戏分包可用格式。
5. 增加触摸区域映射和适配安全区。
